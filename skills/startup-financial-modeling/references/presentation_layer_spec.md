## 1. 読み手と読み順

### 1.1 読み手（3種類。すべて同じ1冊を読む）

| 読み手 | 持ち時間 | 下す判断 | 求める証拠 |
|---|---|---|---|
| **投資家**（非モデラー含む） | 3分でp.1、30分で全体 | 入るか／いくらで入るか／どこで死ぬか | 到達点・必要資金・単位あたり採算・リターン・**弱点の自己申告** |
| **事業側の現場担当**（前提の持ち主） | 10分 | 自分が答えを持つ行はどれか／その値は正しいか | ドライバー名・単位・根拠タグ・**owner** |
| **モデル監査者**（社内・DD） | 60分 | 逃げ道のない手続きを踏んだか | Step連鎖・原本の一意性・独立経路チェック |

### 1.2 読み順（この順に読めば途中で降りない、という導線を紙面で保証する）

```
サマリー p.1（結論・弱点・守っていること）
   ↓ 争点を確かめたい
前提条件（事業前提＝論争対象。付録は後ろに隔離）
   ↓ 数字の作られ方を辿りたい
売上 → 人員 → 費用 → 損益 → 資金繰り（STEP 2→6 の一方向）
   ↓ 水準を確かめたい
KPI → 資本政策 → バリュエーション
   ↓ 逃げ道がないことを確かめたい
各シート末尾の検算ブロック（印刷面）／ 印刷範囲外の監査証跡（xlsx のみ）
```

**原則**: 読み手は必ず「上流シートの結論行」だけを入力として次のシートに進める。上流の**明細**を理解しないと下流が読めない構造を作らない。

## 2. シート別の「唯一の問い」と標準ブロック順

**シートは10枚固定**（`SKILL.md:10-11`）。分割・追加は禁止。列意味が変わる問題はシート分割ではなく**改ページと per-sheet `print_title_rows`** で解く（→ §9）。

### 2.1 重複排除の原則（原本は1か所）

1. **1指標＝1原本行**。原本はその指標を最初に計算するシート（例: 売上高＝売上計画、EBITDA＝損益計画、期末現金＝資金繰り、回収期間＝費用計画、採用レンジ＝バリュエーション）。
2. 他シートの同一指標は**純リンク（緑字）のみ**。再導出（同じ値を別の式で作り直すこと）を禁止する。
3. すべての再掲行は根拠タグ列に `（再掲）原本: <シート>!<ブロック>` を自動出力する。DSLは行に `origin: true | <ref>` を必須で持つ。
4. **セクション見出し文・備考文をシート間でコピーしない**。入力面（前提条件）の見出しは名詞（ドライバー名）、計算面の見出しは動詞（「台数を積む」「費用を積む」）。
5. **同一事実の二重入力を禁止**（例: 「ソース記載の最終年売上」と「Baseケースの最終年売上」を2つの青字セルで持つ）。片方は必ず参照に変換する。

### 2.2 シート別「唯一の問い」と標準ブロック順

各シートは **問い1つ・結論行1つ（headline）** を持つ。ブロック順は上から下へ因果順。`［監査］`は印刷範囲外。

| # | シート | 唯一の問い | headline |
|---|---|---|---|
| S0 | サマリー | この事業は何を作り、いくら要り、どこで死ぬのか | 最終年の主要指標 |
| S1 | 前提条件 | この計画はどの数字を信じろと言い、まだ弱いのはどれか | （なし。入力面） |
| S2 | 売上計画 | 数量はどう積み上がり、いくらの売上になるか | 売上高合計 |
| S3 | 人員計画 | その数量を捌くのに何人要り、人件費はいくらか | 人件費合計 |
| S4 | 費用計画 | 作る・動かす・売るのにいくらかかり、1単位はいつ元が取れるか | 営業費用合計＋回収期間 |
| S5 | 損益計画 | いつ黒字になるか | EBITDA |
| S6 | 資金繰り | 現金はいつ尽きるか | 期末現金 |
| S7 | KPI | 同ステージの標準に届いているか | 要説明項目の一覧 |
| S8 | 資本政策 | 調達後に誰が何%持つか | 希薄化後の創業者持分 |
| S9 | バリュエーション | いくらの値段が付き、いま入る投資家は何倍で返るか | 採用レンジ |

**標準ブロック順（全シート共通の骨格）**

```
1  行2 タイトル帯（右端に短縮凡例）
2  行3 メッセージ行（このページの主張1文・数字を1つ以上含む）
3  行4 STEP行（決めること／使う入力／出す結論）        ← intro()
4  行6 期間ヘッダー（単位列・期間列・根拠タグ列の見出し）
5  本体ブロック（因果順。各ブロック先頭に explain 1行）
6  headline 行（1シート1本・Accent）
7  検算サマリー（1〜3行。件数と全通過の宣言のみ）      ← 印刷面
8  ［監査］チェック明細（独立再計算・行の一致確認・境界ガード）← 印刷範囲外
```

### 2.3 サマリー（S0）— 決定変数で構成する

```
p.1  ① 結論1文 ＋ 到達点3値（最終年売上／黒字化年度／累計調達額）  ★Accent=1か所
     ② 損益・現金ハイライト（金額のみ）
     ③ 単位あたり採算（1単位貢献利益・回収期間）           原本: 費用計画
     ④ 投資家リターン（採用レンジ・MOIC・IRR・希薄化後持分）  原本: バリュエーション/資本政策
     ⑤ この計画が崩れる条件（弱い前提 上位5本：前提名／根拠タグ／検証方法／担当）
     ⑥ この模型が守っていること（4分類×件数×全通過）
     ⑦ いま点灯している警告（名前／該当年／意味。件数だけの表示は禁止）
p.2  ⑧ ソース記載値との照合 ／ ⑨ ケース比較
［監査］ チェック明細全件
```
③④は現状サマリーに存在しない。**投資判断の変数がp.1に1つも載っていない**のが最も高くつく欠陥であり、⑤⑦は「不都合な事実は必ず点灯させる」（`SKILL.md:143`）を件数ではなく中身で果たす。

### 2.4 前提条件（S1）— ブロック順（因果順・番号は連番・欠番禁止）

```
0  ケース選択（3行のみ）: アクティブケース（ドロップダウン）／■必達 スイッチ整合／□要説明 非Base検知
B1 需要（数量）      B2 単価        B3 収益認識      B4 原価（1単位あたり）
B5 人員（連動/計画）  B6 営業費用    B7 投資・運転資本・税
B8 資金調達          B9 資本政策    B10 バリュエーション
──── 以下は付録（事業の前提ではなく採点基準・監査設定。印刷面の最後） ────
A  KPI判定しきい値    B  感応度の振れ幅
C  ソース記載値との照合   D  チェック許容誤差   ［D は印刷範囲外］
```
**ケース持ちドライバーは、そのドライバーが属する事業ブロックの中で**「親ラベル → └Base → └Upside → └Downside → 採用値」の4行小ブロックとして描く（`build_workbook.py:610-611,631` の引き抜きを廃止）。§0はスイッチ3行に縮む。
**許容誤差を最上段に置かない**（現状は事業の話が始まる前に丸め誤差5行が来る）。

### 2.5 計算シートのブロック順（要点のみ）

- **売上計画**: 数量のつながり（期首＋増−減＝期末）→ 顧客・拠点 → 単価×数量 → **売上高合計★** →［監査］
- **人員計画**: 人員（連動型／計画型を行ごとに宣言）→ 人件費 → **人件費合計★** → 実務ガード（獲得キャパ余力）→［監査］
- **費用計画**: 変動費 → 営業費用 → 設備の残高（期首＋取得−償却−除却＝期末）→ 借入の残高 → 費用合計 → **単位あたり採算★** →［監査］
- **損益計画**: **金額の階段だけを連続させる**（売上→原価→粗利→S&M/R&D/G&A→**EBITDA★**→償却→営業利益→営業外→税→純利益）。**比率行を階段の間に挿し込まない**。%はすべて直下の「利益率・費用比率」ブロックに集約する。→ 繰越欠損金のつながり →［監査］
- **資金繰り**: 利益→現金（間接法）→ 投資・財務 → 現金のつながり → **期末現金★** → ランウェイ → 企業価値算定用FCF（原本）→［監査］
- **KPI**: 成長性・収益性 → 効率性 → 資金効率 → **要説明項目の一覧★** →［改ページ］感応度グリッド →［監査］
- **資本政策**: ラウンド前提 → 株主構成のつながり → **希薄化後持分★** → 実務帯の判定 →［監査］
- **バリュエーション**: 手法①DCF（FY展開＋レンジ）→［改ページ］手法②類似上場 ／ 手法③類似取引 →［改ページ］**採用レンジ★**＋投資家リターン →［改ページ］感応度 →［監査］

## 3. モデリングステップの可視化

### 3.1 2軸の番号体系（混ぜない）

- **シートSTEP `S0〜S9`**: 読み順＝因果順。全シートの行4に必ず出す。
- **入力ブロック番号 `B1〜B10`**: ドライバーツリーの節。**前提条件シートでのみ使う**。
  計算シートにドライバーツリーの節番号を持ち出すことを禁止する（現状「4.→6.→7.→8.」と欠番して見え、読み手に無意味な探索を強いる）。計算シートは自分が使う入力ブロックを `使う入力:` で名指しする。

### 3.2 行4のSTEP行（`intro()` を全10シートで必ず呼ぶ）

```
STEP 4/9｜このシートが決めること: 数量を作り・動かし・売るコスト
使う入力: 前提条件 B4・B6・B7 ＋ 売上計画（数量）＋ 人員計画（人件費合計）
出す結論: 売上原価合計・営業費用合計・単位あたり回収期間
```
- `decides` / `uses` / `produces` は DSL 必須。欠落は **ビルドエラー**。
- `uses` に書かれていないシートを参照する数式が存在したら **defect**（宣言と配線の一致検査）。これが「逃げ道のなさ」を機械で保証する唯一の手段。

### 3.3 行3のメッセージ行（`explain` の上位版）

各シートに1文（40〜60字・断定形・数字を1つ以上含む）。**現状の行3は10シートとも完全同一の色凡例**で、17ページに同じ注記を刷っている。ここを空けないと「1ページ1メッセージ」は物理的に置けない。
[SPEC変更] `ib_format_spec §2`「B3に単位・符号規約の注記」→ **B3はメッセージ行**。単位・符号・色凡例はタイトル帯（行2）の右端に短縮版で1回だけ置く。

### 3.4 各セクションの explain（`section(explain=)` を必須引数に昇格）

見出し直下に**専門用語ゼロで40字以内**の1行。例:「期首の台数に新しく動き出した分を足し、退役を引くと期末になる」。
禁止語（§6）を含む explain は**ビルドエラー**。欠落も**ビルドエラー**。

### 3.5 サマリーの「手順地図」

サマリー p.2 に、S0→S9 × [何を決めるか／どのシート／主な入力／出す結論／チェック本数] の表を1つ置く。**この連鎖こそが逃げ道のなさの証拠**であり、現状はエンジン内に存在するのに紙面のどこにも現れていない。

## 4. 視線誘導の規則（3階層）

**大原則: IBのフォント色コード（青=入力／黒=同一シート数式／緑=他シート純リンク／グレー斜体=注記）は一切変更しない。** これ自体が「入力→計算→リンクの一方向」を可視化する装置であり、逃げ道のなさの証拠だからである。Act の色は **IBが色を規定していない層（タイトル帯・強調罫線・余白・行間）にだけ**適用する。

| 階層 | 何 | 書式 | 予算 |
|---|---|---|---|
| **1. 結論** | headline（1シート1本） | 太字＋上細罫線＋下二重罫線＋**Accent `#ECC85A` の下罫線** | **1シート1か所**（2か所以上でビルドエラー） |
| **2. 小計・合計** | 各ブロックの締め | 太字＋上細罫線 | 本体行の**15%以内**（超過は warning） |
| **3. 入力** | 青字＋根拠タグ | フォント色のみ（IB規約） | — |
| **3'. 採用値** | ケース小ブロックの結論 | 太字＋上細罫線（`モデルはこの行のみを参照` を根拠タグに） | ブロック1本 |
| **4. 検算** | チェック行 | ラベル頭に `■必達` / `□要説明`、異常時のみ赤字CF | 印刷面は集計行のみ |

**禁止**
- **背景フィルによる強調を禁止**（`inspect_workbook.py` の `gate(fills == 0, "フィルはタイトル帯のみ")` に抵触。Accentは**罫線**で表現する）。
- 色だけで意味を伝えない（`DESIGN.md`）。`■必達`/`□要説明`、根拠タグの `【記載】【逆算】【ベンチマーク】【仮置き】` という**文字グリフ**を必ず併記する。グレースケール印刷・スライド抽出で情報が消えてはならない。
- 太字を「合計行」と「採用値行」に無差別に付けない（現状は強調が飽和し、視線誘導として機能していない）。

タイトル帯のネイビーは Act Navy `#1F3A66` に統一する（現行 `#1F3864`）。[SPEC変更] `ib_format_spec §1` の `1F3864` と `inspect_workbook.py` の `NAVY` 定数を同一パスで更新。

### 4-1. 点灯した警告は「ラベルごと」赤字にする

サマリーの警告レジスタは、点灯の有無にかかわらず全件を並べる（監査証跡として正しい）。
だが**0と1が同じ黒字で20行並ぶと、読み手は「どれが鳴っているか」を目で探すことになる**。
実測: ATOMで21行中4行が点灯 → 条件付き書式なし → 不都合な事実が灰色の壁に埋もれた。

→ 点灯行は `D:F`（ラベル〜値）に `FormulaRule($F$row>0)` で**赤字太字**。
視線が先に行くのは数字ではなく**行の色**。件数だけを赤くしても足りない。

### 4-2. 列幅は「役割ごとの標準」。見出しは列を広げず折り返す

| 列 | 役割 | 標準幅 |
|---|---|---|
| A | スペーサー | 2 |
| B・C | 階層インデント | 2.14 |
| D | ラベル | 52 |
| E | 単位 | 9 |
| F〜 | **数値（期間・イベント）** | **11.5 固定** |
| （補助） | ベンチマーク等の文章列 | 12〜24 |
| 末尾 | 根拠タグ | 16 |

**数値列を見出しに合わせて広げてはならない。** 旧実装は `幅 = min(見出し長+1.5, 22)` で自動拡張しており、
ラウンド名の長い資本政策が**幅227**まで膨らんだ。印刷は `fitToWidth=1` なので、
1列でも広いと**シート全体が縮小され**、標準シート（幅141→縮尺68%）に対して**縮尺42%**＝
数字が読めない大きさで刷られる。見出しは `wrap_text` で折り返し、見出し行（行5）だけ高さを持たせる。
（幅の合計を定数で縛るのではなく「列は役割どおりの幅か」を問う。期間列の本数が違うシートにそのまま効く。）

## 5. 行間・ブロック規則

**行高のカスタム指定は禁止**（`inspect_workbook.py:83` のゲート・`ib_format_spec §2`・IB作法）。したがって行間を作れるレバーは**空行数**と**印刷縮尺**の2つしかない。

### 5.1 空行の文法（`blank(n)` に拡張）

| 境界 | 空行 |
|---|---|
| トップレベルのセクション境界（＝ページの話題が変わる） | **2行**（＋改ページ） |
| ドライバー家族／サブブロックの境界 | **1行** |

| 空行3行以上 | **禁止**（用紙都合の隙間は改ページで解く） |

[SPEC変更] `ib_format_spec §2`「セクション間は空行1行」→ 上表で置き換える。

### 5.2 空行の文法は縮尺の回復とセットでしか効かない

現状の印刷可視幅は A〜K で約233文字。横1ページ（`fitToWidth=1`）に収めるため縮尺は40〜55%に落ち、**Arial 10pt が実効5pt台**、空行1行は実効2〜3ptで**視認不能**になる。
→ §9の「備考列を印刷範囲外へ」を実施して縮尺を90%台に戻して初めて、空行の文法が読み手に届く。**この2つは必ずセットで実装する。**

## 7. Assumptionシートの設計方法論（MECE分解）

### 7.1 「現場解像度が高い」の定義（＝オーナー可能な粒度）

数字は細かければ現場的なのではない。次の**3条件を同時に**満たすとき、その変数は現場解像度が高い。

1. **単独オーナー**: 社内にその数字を守る責任者が**ちょうど1人**いる。0人なら抽象すぎる（「売上成長率50%」は誰も持てない）。2人以上なら未分解（「人件費」は採用責任者と報酬責任者の2人が持つ → 「人数」×「単価」に割る）。
2. **観測可能な現物**: その値が**今日**、実在の紙・画面から読める（見積書・雇用契約・料金表・稼働ログ・求人票・請求書）。他の計画値から逆算した数は現物ではない。
3. **今期に動かせる**: オーナーが四半期内の自分の行動で値を変えられる（採用する／値上げする／台数を積む）。動かせないならそれは前提ではなく制約。

### 7.2 分解の手順（この順にしか進まない。前提条件のブロック順と一致する）

```
1 単位を決める     何を1つ売ると1つ売上が立つか（誤ると以降すべて崩れる）
2 数量を積む       期首＋増加−減少＝期末。増加と減少に別のオーナーを付ける
3 単価を置く       数量とは別の行にする。掛けずに並べる
4 収益＝単価×数量   成長率は結果であって入力ではない（入力に成長率が現れたら分解失敗）
5 原価             1単位あたりで書く（1単位製造原価・1単位運用費）
6 人員             「需要量÷1人あたりキャパ（連動型）」か「意思決定人数（計画型）」を行ごとに宣言
7 営業費用         率×ベース。ベースは既にモデル内に存在する行でなければならない
8 投資・運転資本・税
9 資金・資本
```

### 7.3 分解パターン（どの事業もこの6型のいずれかに当てはめる。DSL必須フィールド `pattern`）

| 型 | 標準の行並び（上→下） |
|---|---|
| ストック・フロー | 期首 → ＋増加 → −減少 → ＝期末 |
| 価格×数量 | 数量 → 単価 → ＝収益 |
| ファネル | 母集団 → 転換率（段階ごとに1行）→ 件数 → ×単価 |
| 人員連動 | 需要量 → 1人あたりキャパ → ＝必要人員 → ×単価 |
| 率×ベース | ベース行（既存行への参照）→ 率 → ＝金額（**率の上に率を重ねない**） |
| 単価×件数 | 件数（参照）→ 1件あたり単価 → ＝金額 |

すべてのコスト行は **変動（数量連動）／準変動（人員連動）／固定（意思決定）のちょうど1つ**に属する（DSL必須フィールド `cost_class`）。分類の合計＝損益計画の費用合計 を**検算行で機械検証**する。これがMECE性の唯一の証明であり、そのまま感応度分解の入力になる。

### 7.4 停止条件（どれか1つでも当たったら分解を止める）

| | 条件 |
|---|---|
| S1 | オーナーが1人になった（2人いる間は割り続ける） |
| S2 | 子の値が現物から読める（見積書・契約・料金表）。それ以上割ると現物が消える |
| S3 | 3ケースが**全期間**同値になった＝シナリオ変数ではなく定数。`cases:` を外す（初年度だけ同値のケースは対象外） |
| S4 | 親が【仮置き】なのに子も【仮置き】＝推測を2つに増やしただけ。証拠がある最上位で止める |
| S5 | 最終年EBITDAへの感応度が閾値未満（±10%振って影響1%未満）＝答えを動かさない行 |
| S6 | 1ドライバー家族の入力行が7行を超えた＝前提条件ではなく専用計算シートの仕事 |

S5は**静的検査では判定不能**。`scenario_sweep.py` による再計算が必要（→ §10で「非静的」として分離する）。

### 7.5 ドライバーの必須フィールド（assumption register スキーマ拡張）

既存: `id / label / unit / fmt / kind / basis / source / values / cases`
**追加（欠落はビルドエラー）**:

| フィールド | 内容 |
|---|---|
| `pattern` | §7.3 の6型のいずれか |
| `cost_class` | 変動／準変動／固定（コスト行のみ） |
| `owner` | その数字を守る職能（例: 調達／遠隔オペ責任者／人事） |
| `verification` | どうすれば仮置きが実績に変わるか（例: 初期10台の稼働ログ3か月） |

`basis ∈ {仮置き, 逆算, ベンチマーク}` の行は `verification` と `owner` の記入を**必須**とし、サマリー⑤「この計画が崩れる条件」に自動で吸い上げる。**投資家が最も評価するのは「どの前提がまだ弱く、いつ・どうやって潰すか」の自己申告**であり、現状これは紙面に一切存在しない。

### 7.6 定型文の圧縮

同一文言の備考が同一ブロック内で3回以上繰り返されたら、ブロック見出しの note に1回だけ書き、各行からは削る（例:「Upside＝×1.4／Downside＝×0.7（需要・体制・投資を連動）」は見出しで1回宣言する）。ケース行の備考は**Baseとの差分の理由だけ**を書く。

## 8. チェック行の備考規則

### 8.1 4フィールド必須（DSL `check()` に追加。欠落はビルドエラー）

| フィールド | 書き方 |
|---|---|
| `guards` | **何を守るか**を平文の式で。「期末の設備＝期首＋購入−償却−除却」 |
| `if_broken` | **破れたら何が嘘になるか**。「買った設備がどこかで消えているか、二重に数えている」 |
| `independent` | **なぜ独立経路か**。「入力の取得単価×台数から作り直しており、本体行のセルを一切読まない」 |
| `tol` | 許容誤差（末尾に小さく） |

備考の自動生成形:
> 守る式: 期末の設備＝期首＋購入−償却−除却。／ 破れたら: 買った設備がどこかで消えているか、二重に数えている。／ 独立経路: 入力の取得単価×台数から再導出（本体行を読まない）。／ 許容誤差 ±0.5

現状は**164行中、平文で意味を説明した注記が0本**（許容誤差のみ／汎用文の使い回し／恒等式の記号提示）。守りの厚さがそのまま読みにくさに変換されている。同じ164本が、この書式で**信頼の証拠**に変わる。

### 8.2 等級を紙面に描く（クラス表示は `cls` から導出する。手打ち禁止）

`cls=error` → ラベル頭に `■必達`、`cls=alert` → `□要説明`。

**この前置きを呼び出し側の手打ちに任せてはならない。** `check()` の内部で `cls` から必ず付与する
（既存の前置きは剥がしてから付け直す）。手打ちに任せると次の2つが必ず起きる:

1. **付け忘れ** — 非負チェック5本がマーカー無しで出荷され、紙面上**導出行と区別できなかった**。
2. **ラベルと実体の食い違い** — 「■必達」と書いてあるのに `cls=alert` で登録されており、
   破れても総合判定が鳴らない（＝最悪の偽の保証）。

機械ゲート: `（0=OK）` で終わる行に `■`/`□` が無ければ FAIL。

### 8.3 ラベルの一意性

チェックラベルはブック内で一意。衝突したら由来行の修飾子（括弧内）を保持して曖昧性を解消する（現状、同名ラベルが3行並び、NG時に原因行へ辿り着けない）。truncation 禁止。

### 8.4 見せ方の反転（証拠を証拠として機能させる）

- **明細は全10シートで印刷範囲外**（現状サマリーだけが守っている。他9シートは `build_workbook.py:2748` の `setdefault(ws.title, ws.max_row)` で行タイ生成後の最終行を拾い、1ページ丸ごと `（0=OK）` の羅列になっている）。
- **印刷面に残すのは各シート末尾の検算サマリー1〜3行**:
  `独立再計算 6項目 全一致 ／ 閉合 3項目 全一致 ／ 実務ガード 2項目（うち警告1）`
- **非負ガード等を本文行の間に挿入することを禁止**（事業の連鎖の途中で読み手の視線が切れる）。ガードもシート末尾の検算ブロックに集める。
- **サマリーに「この模型が守っていること」を印刷面で新設**: ①配線の一致 ②独立経路の再計算 ③境界の妥当性 ④ソース記載値との照合 の4分類 × 件数 × 全通過。
- 明細は `outlineLevel=1` で既定折りたたみ（印刷範囲外に置くだけでは、Excelを下にスクロールした読み手に暗号が地続きで現れる）。

§8.4（明細を隠す）と §8.1〜8.3（意味を書く）は**必ずセット**で実装する。片方だけだと「逃げ道のなさを見せる」という要求と矛盾する。

### 8.5 導出行の備考は「検証を名乗ってはならない」（逃げ道の封じ込め）

もっとも危険な逃げ道。実例:

```
売上計画!12  新規出荷・稼働開始台数 = (期末の目標台数 − 期首) + 退役
             備考: 「ロールフォワード閉合: 期首＋新規−退役＝期末」
```

出荷台数を**期末の目標から逆算**しておきながら、備考は「閉合」と称している。
この等式は**逆算した瞬間に定義上必ず成り立つ**。何も検証していないのに、
読み手は「検算が通った行」と受け取る。**プラグ（逆算）が、検算の権威を借りている。**

規則:

- **検証を名乗れるのはチェック行だけ。** 導出行の備考に
  `閉合` `検算` `整合確認` `一致確認` `検証済` を書いてはならない（機械ゲート）。
- 逆算行は**逆算だと名乗る**。「期末の目標台数から逆算。等式は定義上必ず成り立つので
  正しさの根拠にならない。製造・出荷能力の裏付けはモデル外」
- 節見出し（数式を持たない行）から検算行を**指し示す**のは正当（「残高の一致は検算行で確かめる」）。
  ゲートは数式を持つ行にだけ効かせる。
